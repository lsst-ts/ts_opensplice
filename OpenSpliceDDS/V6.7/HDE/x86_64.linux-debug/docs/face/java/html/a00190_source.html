102 
103 This QoS policy controls the extent to which changes to data-instances can be made dependent on each other,
104 the order in which they need to be presented to the user and also the kind of dependencies that can be propagated
105 and maintained by the Service.
106 
107 The support for ‘coherent changes’ enables a publishing application to change the value of several data-instances that
108 could belong to the same or different topics and have those changes be seen ‘atomically’ by the readers. This is useful
109 in cases where the values are inter-related. For example, if there are two data-instances representing the ‘altitude’
110 and ‘velocity vector’ of the same aircraft and both are changed, it may be useful to communicate those values in a way
111 the reader can see both together; otherwise it may erroneously interpret that the aircraft is on a collision course.
112 Basically this QosPolicy allows a \ref DCPS_Modules_Publication "Publisher" to group a number of samples by writing them
113 \if isocpp2
114 within a scope that holds a CoherentSet object
115 \else
116 within calls to begin_coherent_change and end_coherent_change
117 \endif
118 and treat them as if they are to be communicated as a single message. That is, the receiver will only be able to access the data after all
119 the modifications in the set are available at the receiver end.
120 
121 Samples that belong to a (yet) unfinished coherent update consume resource limits from the receiving \ref DCPS_Modules_Subscription_DataReader "DataReader", but are not
122 (yet) accessible through its history, and cannot (yet) push samples out of its history. In order for the \ref DCPS_Modules_Subscription_DataReader "DataReader" to store
123 samples outside its history administration, its ResourceLimitsQosPolicy should have a value for max_samples_per_instance that
124 is bigger than the depth value of its HistoryQosPolicy.
125 
126 If not enough resources are available to hold an incoming sample that belongs to an unfinshed transaction, one of the following
127 things may happen.
128 - When some of the resources are in use by the history of the \ref DCPS_Modules_Subscription_DataReader "DataReader", the incoming sample will be rejected, and the \ref DCPS_Modules_Subscription_DataReader "DataReader"
129 will be notified of a SAMPLE_REJECTED event. This will cause the delivery mechanism to retry delivery of the rejected sample at a
130 later moment in time, in the expectation that the application will free resources by actively taking samples out of the reader history.
131 - When all the available resources are in use by samples belonging to unfinished coherent updates, the application has no way to free
132 up resources and the transaction is either ‘deadlocked’ by itself (i.e. it is too big for the amount of available resources) or by one
133 or more other incomplete transactions. To break out of this deadlock, all samples belonging to the same transaction as the currently
134 incoming sample will be dropped, and the \ref DCPS_Modules_Subscription_DataReader "DataReader" will be notified of a SAMPLE_LOST event. No attempt will be made to retransmit
135 the dropped transaction. To avoid this scenario, it is important to make sure the \ref DCPS_Modules_Subscription_DataReader "DataReader" has set its ResourceLimits to accommodate
136 for the worst case history size PLUS the worst case transaction size. In other words, if Sh represents the worst case size of the
137 required history, St represents the worst case size of single transaction and Nt represents the worst case number of concurrent
138 transactions, then the ResourceLimits should accomodate for Sh + (St * Nt).
139 
140 A connectivity change may occur in the middle of a set of coherent changes; for example, the set of partitions used by the \ref DCPS_Modules_Publication "Publisher" or
141 one of its Subscribers may change, a late-joining \ref DCPS_Modules_Subscription_DataReader "DataReader" may appear on the network, or a communication failure may occur. In the
142 event that such a change prevents an entity from receiving the entire set of coherent changes, that entity must behave as if it had
143 received none of the set.
144 
145 The support for ‘ordered_access’ enables a subscribing application to view changes in the order in which they occurred. Ordering is always
146 determined according to the applicable DestinationOrderQosPolicy setting. Depending on the selected access_scope, ordering is either on a
147 per instance basis (this is the default behaviour, even when ordered_access is set to FALSE), on a per \ref DCPS_Modules_Subscription_DataReader "DataReader" basis or across all
148 DataReaders that span the \ref DCPS_Modules_Subscription "Subscriber". In case of ordered_access with an acces_scope of GROUP, the \ref DCPS_Modules_Subscription "Subscriber" will enforce that all its
149 DataReaders share the same DestinationOrderQosPolicy setting. The DestinationOrderQosPolicy setting of the first \ref DCPS_Modules_Subscription_DataReader "DataReader" created for
150 that \ref DCPS_Modules_Subscription "Subscriber" will then determine the DestinationOrderQosPolicy setting that is allowed for all subsequent DataReaders. Conflicting
151 settings will result in an INCONSISTENT_POLICY error.
152 
153 The PresentationQosPolicy is applicable to both \ref DCPS_Modules_Publication "Publisher" and \ref DCPS_Modules_Subscription "Subscriber", but behaves differently on the publishing side and the subscribing
154 side. The setting of coherent_access on a \ref DCPS_Modules_Publication "Publisher" controls whether that \ref DCPS_Modules_Publication "Publisher" will preserve the coherency of changes
155 \if isocpp2
156  within a scope that holds a CoherentSet object
157 \else
158  enclosed by calls to begin_coherent_change and end_coherent_change
159 \endif
160 , as indicated by its access_scope and as made available by its embedded DataWriters. However, the \ref DCPS_Modules_Subscription "Subscriber" settings determine whether a coherent set of samples will actually be delivered to the subscribing application in a coherent way.
161 
162 - If a \ref DCPS_Modules_Publication "Publisher" or \ref DCPS_Modules_Subscription "Subscriber" sets coherent_access to FALSE, it indicates that it does not want to maintain coherency between the different
163 samples in a set: a \ref DCPS_Modules_Subscription "Subscriber" that receives only a part of this set may still deliver this partial set of samples to its embedded DataReaders.
164 - If both \ref DCPS_Modules_Publication "Publisher" and \ref DCPS_Modules_Subscription "Subscriber" set coherent_access to TRUE, they indicate that they want to maintain coherency between the different samples
165 in a set: a \ref DCPS_Modules_Subscription "Subscriber" that receives only a part of this set may not deliver this partial set of samples to its embedded DataReaders; it needs
166 to wait for the set to become complete, and it will flush this partial set when it concludes that it will never be able to complete it.
167 
168 Coherency is implemented on top of a transaction mechanism between individual DataWriters and DataReaders; completeness of a coherent set is
169 determined by the successful completion of each of its participating transactions. The value of the access_scope attribute determines which
170 combination of transactions constitute the contents of a coherent set.
171 
172 The setting of ordered_access has no impact on the way in which a \ref DCPS_Modules_Publication "Publisher" transmits its samples (although it does influence the RxO properties
173 of this \ref DCPS_Modules_Publication "Publisher"), but basically it determines whether a \ref DCPS_Modules_Subscription "Subscriber" will preserve the ordering of samples when the subscribing application uses
174 its embedded DataReaders to read or take samples:
175 - If a \ref DCPS_Modules_Subscription "Subscriber" sets ordered_access to FALSE, it indicates that it does not want to maintain ordering between the different samples it receives:
176 a subscribing application that reads or takes samples will receive these samples ordered by their key-values, which does probably not resemble the
177 order they were written in.
178 - If a \ref DCPS_Modules_Subscription "Subscriber" sets ordered_access to TRUE, it indicates that it does want to maintain ordering within the specified access_scope between the
179 different samples it receives: a subscribing application that reads or takes samples will receives these samples sorted by the order in which they
180 were written.
181 
182 The access_scope determines the maximum extent of coherent and/or ordered changes:
183 - If access_scope is set to INSTANCE and coherent_access is set to TRUE, then the \ref DCPS_Modules_Subscription "Subscriber" will behave, with respect to
184 maintaining coherency, in a way similar to an access_scope that is set to TOPIC. This is caused by the fact that coherency
185 is defined as the successful completion of all participating transactions. If a \ref DCPS_Modules_Publication_DataWriter "DataWriter" writes a transaction containing samples from different
186 instances, and a connected \ref DCPS_Modules_Subscription_DataReader "DataReader" misses one of these samples, then the transaction failed and the coherent set is considered incomplete by the
187 receiving \ref DCPS_Modules_Subscription_DataReader "DataReader". It doesn’t matter that all the other instances have received their samples successfully; an unsuccessful transaction by
188 definition results in an incomplete coherent set. In that respect the DDS can offer no granularity that is more fine-grained with respect to
189 coherency than that described by the TOPIC.
190 If access_scope is set to INSTANCE and ordered_access is set to TRUE, then the \ref DCPS_Modules_Subscription "Subscriber" will maintain ordering between
191 samples belonging to the same instance. Samples belonging to different instances will still be grouped by their key-values instead of by the order
192 in which they were received.
193 - If access_scope is set to TOPIC and coherent_access is set to TRUE, then the DDS will define the scope of a coherent set
194 on individual transactions. So a coherent set that spans samples coming from multiple DataWriters
195 \if isocpp2
196  (indicated by the scope of the CoherentSet object),
197 \else
198  (indicated by its enclosure within calls to begin_coherent_change and end_coherent_change on their shared \ref DCPS_Modules_Publication "Publisher"),
199 \endif
200 is chopped up into separate and disjunct transactions (one for each participating \ref DCPS_Modules_Publication_DataWriter "DataWriter"), where each transaction is processed separately.
201 On the subscribing side this may result in the successful completion of some of these transactions, and the unsuccessful completion of some others.
202 In such cases all DataReaders that received successful transactions will deliver the embedded content to their applications, without waiting for the
203 completion of other transactions in other DataReaders connected to the same \ref DCPS_Modules_Subscription "Subscriber".
204 If access_scope is set to TOPIC and ordered_access is set to TRUE, then the \ref DCPS_Modules_Subscription "Subscriber" will maintain ordering between samples
205 belonging to the same \ref DCPS_Modules_Subscription_DataReader "DataReader". This means that samples belonging to the same instance in the same \ref DCPS_Modules_Subscription_DataReader "DataReader" may no longer be received consecutively
206 if samples belonging to different instances were written in between. It is possible to read/take a limited batch of ordered samples (where max_samples
207 != LENGTH_UNLIMITED). In that case the \ref DCPS_Modules_Subscription_DataReader "DataReader" will keep a bookmark, so that in subsequent read/take operations your application can start where the
208 previous read/take call left off. There are two ways for the middleware to indicate that you completed a full iteration:
209  + The amount of samples returned is smaller than the amount of samples requested. In that case the bookmark is automatically reset and the next
210 read/take will begin a new iteration right from the start of the ordered list.
211  + Your read/take call returns RETCODE_NO_DATA, indicating that there are no more samples matching your criteria after the current bookmark. In that
212 case the bookmark is reset as well.
213 The bookmark is also reset in the following cases:
214  + A read/take call uses different masks than the previous invocation of read/take.
215  + A read/take call uses a different query than the previous invocation of read/take.
216 - If access_scope is set to GROUP and coherent_access is set to TRUE, then the DDS will define the scope of a coherent set on the
217 sum of all participating transactions. So a coherent set that spans samples coming from multiple DataWriters
218 \if isocpp2
219  (indicated by the scope of the CoherentSet object),
220 \else
221  (indicated by its enclosure within calls to begin_coherent_change and end_coherent_change on their shared \ref DCPS_Modules_Publication "Publisher"),
222 \endif
223 is chopped up into separate and disjunct transactions (one for each participating \ref DCPS_Modules_Publication_DataWriter "DataWriter"), where each transactions is processed separately. On the
224 subscribing side this may result in the successful completion of some of these transactions, and the unsuccessful completion of some others. However,
225 each \ref DCPS_Modules_Subscription_DataReader "DataReader" is only allowed to deliver the embedded content when all participating transactions completed successfully. This means that DataReaders
226 that received successful transactions will need to wait for all other DataReaders attached to the same \ref DCPS_Modules_Subscription "Subscriber" to also complete their transactions
227 successfully. If one or more DataReaders conclude that they will not be able to complete their transactions successfully, then all DataReaders that
228 participate in the original coherent set will flush the content of their transactions. In order for the application to access the state of all
229 DataReaders that span the coherent update, a separate read/take operation will need to be performed on each of the concerned DataReaders. To keep the
230 history state of the DataReaders consistent in between the successive invocations of the read/take operations on the various readers,
231 \if isocpp2
232  the DataReaders should be locked for incoming updates by starting a new scope that holds a CoherentAccess object.
233 If all concerned DataReaders have been accessed properly, they can be unlocked for incoming updates by ending the scope that holds the CoherentAccess object or by explicitly invoking its operation named "end".
234 \else
235  the DataReaders should be locked for incoming updates by invoking the begin_access on the \ref DCPS_Modules_Subscription "Subscriber" prior to accessing the first \ref DCPS_Modules_Subscription_DataReader "DataReader".
236 If all concerned DataReaders have been accessed properly, they can be unlocked for incoming updates by invoking the end_access on the \ref DCPS_Modules_Subscription "Subscriber".
237 \endif

239 If access_scope is set to GROUP and ordered_access is set to TRUE, then ordering is maintained between samples that are written
240 by DataWriters attached to a common \ref DCPS_Modules_Publication "Publisher" and received by DataReaders attached to a common \ref DCPS_Modules_Subscription "Subscriber". This way the subscribing application can
241 access the changes as a unit and/or in the proper order. However, this does not necessarily imply that the subscribing application will indeed access
242 the changes as a unit and/or in the correct order. For that to occur, the subscribing application must use the proper logic in accessing its datareaders:
243  + Upon notification by the callback operation on_data_on_readers of the SubscriberListener or when triggered by the similar DATA_ON_READERS status of
244 the \ref DCPS_Modules_Subscription "Subscriber"’s StatusCondition,
245 \if isocpp2
246 the application starts a new scope holding a CoherentAccess object
247 \else
248 the application uses begin_access on the \ref DCPS_Modules_Subscription "Subscriber"
249 \endif
250 to indicate it will be accessing data through the \ref DCPS_Modules_Subscription "Subscriber". This will lock the embedded datareaders for any incoming messages during the
251 coherent data access.
252  + Then it
253 \if isocpp2
254  invokes dds::sub::find function
255 \else
256  calls get_datareaders on the \ref DCPS_Modules_Subscription "Subscriber"
257 \endif
258 to get the list of \ref DCPS_Modules_Subscription_DataReader "DataReader" objects
259 where data samples are available. Note that when ordered_access is TRUE, then the list of DataReaders may contain the same reader several times. In
260 this manner the correct sample order can be maintained among samples in different \ref DCPS_Modules_Subscription_DataReader "DataReader" objects.
261  + Following this it calls read or take on each \ref DCPS_Modules_Subscription_DataReader "DataReader" in the same order returned to access all the relevant changes in the \ref DCPS_Modules_Subscription_DataReader "DataReader". Note that when
262 ordered_access is TRUE, you should only read or take one sample at a time.
263  + Once it has called read or take on all the readers,
264 \if isocpp2
265  it ends the scope that holds the CoherentAccess object or it explicitly invokes its operation named "end"
266 \else
267  it calls end_access on the \ref DCPS_Modules_Subscription "Subscriber".
268 \endif
269 This will unlock the embedded datareaders again.
