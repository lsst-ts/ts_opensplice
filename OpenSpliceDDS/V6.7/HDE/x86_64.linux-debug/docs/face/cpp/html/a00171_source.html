1 SubscriptionKey QoS {#DCPS_QoS_SubscriptionKey}
2 ==================
3 
4 This QosPolicy allows the \ref DCPS_Modules_Subscription_DataReader "DataReader" to define it's own set of keys on the data,
5 potentially different from the keys defined on the \ref DCPS_Modules_TopicDefinition "Topic".
6 
7 *NOTE:* This is an OpenSplice-specific QosPolicy, it is not part of the DDS
8 Specification.

38 
39 By using the SubscriptionKeyQosPolicy, a \ref DCPS_Modules_Subscription_DataReader "DataReader" can force its own key-list
40 definition on data samples. The consequences are that the \ref DCPS_Modules_Subscription_DataReader "DataReader" will
41 internally keep track of instances based on its own key list, instead of the key list
42 dictated by the \ref DCPS_Modules_TopicDefinition "Topic".
43 
44 Operations that operate on instances or instance handles, such as
45 lookup_instance or get_key_value, respect the alternative key-list and work
46 as expected. However, since the mapping of writer instances to reader instances is
47 no longer trivial (one writer instance may now map to more than one matching
48 reader instance and vice versa), a writer instance will no longer be able to fully
49 determine the lifecycle of its matching reader instance, nor the value its
50 view_state and instance_state.
51 
52 In fact, by diverting from the conceptual 1 – 1 mapping between writer instance and
53 reader instance, the writer can no longer keep an (empty) reader instance
54 ALIVE by just refusing to unregister its matching writer instance. That means that when a
55 reader takes all samples from a particular reader instance, that reader instance will
56 immediately be removed from the reader’s administration. Any subsequent
57 reception of a message with the same keys will re-introduce the instance into the
58 reader administration, setting its view_state back to NEW. Compare this to the
59 default behaviour, where the reader instance will be kept alive as long as the writer
60 does not unregister it. That causes the view_state in the reader instance to remain
61 NOT_NEW, even if the reader has consumed all of its samples prior to receiving an
62 update.
63 
64 Another consequence of allowing an alternative keylist is that events that are
65 communicated by invalid samples (i.e. samples that have only initialized their
66 keyfields) may no longer be interpreted by the reader to avoid situations in which
67 uninitialized non-keyfields are treated as keys in the alternative keylist. This
68 effectively means that all invalid samples (e.g. unregister messages and both
69 implicit and explicit dispose messages) will be skipped and can no longer affect the
70 instance_state, which will therefore remain ALIVE. The only exceptions to this
71 are the messages that are transmitted explicitly using the
72 writedispose() call, which always includes a full and valid sample and can therefore modify the
73 instance_state to NOT_ALIVE_DISPOSED.
74 
75 By default, the SubscriptionKeyQosPolicy is not used because use_key_list is set to FALSE.
76 This QosPolicy is applicable to a \ref DCPS_Modules_Subscription_DataReader "DataReader" only, and cannot be changed after the
77 \ref DCPS_Modules_Subscription_DataReader "DataReader" is enabled.
