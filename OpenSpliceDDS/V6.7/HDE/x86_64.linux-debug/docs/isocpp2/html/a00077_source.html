1 ISO/IEC C++ DCPS API {#mainpage}
2 ====================
3 
4 [TOC]
5 
6 ISO C++ is the latest DDS DCPS API for the C++ Language. It is more user friendly
7 and simpler to use than the 'classic' C++ API.
8 
9 This is the second version of the ISOC++ Specification that has been implemented.<br>
10 While most things haven't changed between ISOC++ V1 and V2 regarding the API, there
11 are still some differences that should be considered when migrating from V1 to V2. These
12 differences are addressed in the [Migration Guide] (../../html/isocpp_migration.html).<br>
13 That [Migration Guide] (../../html/isocpp_migration.html) also contains the rationale
14 for the second version of ISOC++.
15 
16 This documentation serves to provide a reference guide and a basic introduction to
17 the ISO C++ API. Due to the way that ISO C++ is implemented using delegates, the
18 API reference is not a 1:1 representation of the implementation structure of the API,
19 it is however a representation of the user accessible layer of the API.

38 How to use ISO C++ classes {#mainpage_howtouse}
39 --------------------------
40 
41 ISO C++ was designed in such a way that all memory management is handled automatically
42 for you. Class instances should be created on the stack rather than the heap and the API itself
43 will handle the allocation and deletion of memory automatically.
44 
45 There are two main types of class within ISO C++, Value and Entity. All classes within
46 ISO C++ inherit from one of these two classes.
47 
48 Value classes are the simplest type of class within ISO C++. These are simple classes which
49 are created on the stack and are copied by value. Examples of this type of class include QoS policies
50 such as dds::core::policy::Durability and dds::core::policy::Liveliness or status conditions
51 such as dds::core::status::LivelinessChangedStatus.
52 
53 Entity classes on the other hand are more advanced but still easy to use. These contain an
54 underlying smart pointer which keeps track of the number of references that exist to that
55 particular instance of the class. Once there are no remaining references to the class instance, any
56 memory used by it is then automatically deleted. This underlying smart pointer brings with it
57 the advantage of low overhead copying of class instances as only the smart pointer needs to be copied.
58 This means that any number of references may point to the same entity class instance.
59 
60 Below is an example of how a dds::domain::DomainParticipant can be created and copied, other
61 Entity classes work in the same way.
62 
63  {
64  dds::domain::DomainParticipant dp1(dds::core::null); //dds::core::null creates an uninitialised Entity
65  {
66  dds::domain::DomainParticipant dp2(0); //Creates a domain participant on domain 0
67 
68  dp1 = dp2; //Copies reference to dp2 into dp1, both references now point to the same class instance
69  }
70  //dp2 is now out of scope but the domain participant is not deleted as dp1 still has a reference to it
71  }
72  //dp1 is now out of scope and there are no further references to the domain participant so it is deleted automatically
73

78 API documentation {#mainpage_api}
79 =================
80 
81 The namespaces below contain all of the major functionality of the ISO C++ API:
82 
83 - ::dds
84 
85  + dds::domain
86  + The dds::domain namespace provides facilities for creating a DomainParticipant
87  that establishes a virtual network in which an application and it's entities
88  can operate.
89  + dds::topic
90  + The dds::topic namespace provides facilities for the creation of Topics which
91  can be published or subscribed to, as well as providing more advanced filtering
92  through ContentFilteredTopics.
93  + dds::pub
94  + The dds::pub namespace provides facilities for writing samples based on a Topic.
95  + dds::sub
96  + The dds::sub namespace provides facilities for reading samples based on a Topic,
97  as well as providing the ability to query a read.
98  + dds::core
99  + The dds::core namespace provides core facilities such as QoS policies and the
100  ability to wait on particular conditions using a WaitSet.

102 Platform requirements & re-building {#mainpage_build}
103 ===================================
104 
105 The ISO C++ PSM specification requires some modern compiler features.
106 Specifically: the header file [ref_traits.hpp](@ref dds/core/ref_traits.hpp)
107 requires that shared and weak smart pointer types and functions for safe
108 polymorphism be available and are used. The header
109 [macros.hpp](@ref dds/core/macros.hpp) requires the existence of a _compile
110 time_ static assert.
111 
112 To address these the OpenSplice ISO C++ DCPS API requires any one only of the
113 following:
114  + A compiler with C++ 11 support
115  + A compiler with Technical Report 1 (TR1) support
116  + The availability of [Boost]. Headers only are required.
117 
118 GCC requirements {#mainpage_gcc}
119 ----------------
120 
121 By default, the ISO C++ DCPS API can, and does, compile against the
122 TR1 templates headers available in g++ since version 4.1. Any
123 distributions built with an older version of GCC will have been compiled
124 against [Boost].
125 
126 If you would like to use your own version of [Boost] or the experimental
127 `-std=c++0x` C++11 support to develop your applications: you should rebuild the
128 ISO C++ library to suit your needs.
129 See [the rebuilding instructions](@ref isocpp2_customlibs)

158 Regarding objects {#mainpage_objects}
159 =================
160 
161 The way the code of the specification is set up is not very intuitive, but this is basically what happens:<br>
162 The dds::sub::Subscriber is a typedef to the dds::sub::TSubscriber&lt;DELEGATE&gt; template class.<br>
163 The dds::sub::TSubscriber&lt;DELEGATE&gt; is basically just a wrapper to the implementation template class (aka delegate) dds::sub::detail::Subscriber.
164 
165 So, when you create a dds::sub::Subscriber, you create the wrapper dds::sub::TSubscriber&lt;dds::sub::detail::Subscriber&gt; to the implementation class dds::sub::detail::Subscriber.
166 
167 Type dependent objects have just one more template argument:<br>
168 The dds::sub::DataReader is a typedef to the dds::sub::TDataReader&lt;T, DELEGATE&lt;T&gt;&gt; template class.<br>
169 The dds::sub::TDataReader&lt;T, DELEGATE&lt;T&gt;&gt; is the wrapper to the delegate dds::sub::detail::DataReader&lt;T&gt;.<br>
170 Thus dds::sub::DataReader&lt;my_type&gt; is a typedef to dds::sub::TDataReader&lt;my_type, dds::sub::detail::DataReader&lt;my_type&gt;&gt; which is an wrapper to class dds::sub::detail::DataReader&lt;my_type&gt;.
171 
172 The API shields these wrappers and delegates from the user by means of the typedefs. The API documentation depicts how these typedefs should be used, e.g. as normal classes and objects.
