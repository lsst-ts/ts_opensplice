6 Built-in Topics {#DCPS_Builtin_Topics_Intro}
7 ===============
8 
9 As part of its operation, the middleware must discover and possibly keep track of
10 the presence of remote entities such as a new participant in the domain. This
11 information may also be important to the application, which may want to react to
12 this discovery, or else access it on demand.
13 
14 To make this information accessible to the application, the DCPS specification
15 introduces a set of built-in topics and corresponding DataReader objects that can
16 then be used by the application. The information is then accessed as normal
17 application data. This approach avoids introducing a new API to access this
18 information and allows the application to become aware of any changes in those
19 values by means of any of the mechanisms presented in \ref DCPS_Modules_Infrastructure_Listener
20 "Listeners" and \ref DCPS_Modules_Infrastructure_Waitset "Conditions and Waitsets".
21 
22 \if isocpp2
23 The built-in data-readers all belong to a built-in Subscriber. This subscriber can be
24 retrieved by using the method dds::sub::builtin_subscriber with the DomainParticipant
25 as parameter.<br>
26 The built-in DataReader objects can be retrieved by using the operation
27 \ref dds::sub::find(const dds::sub::Subscriber& sub, const std::string &topic_name, FwdIterator begin, uint32_t max_size)
28 "dds::sub::find", with the BuiltinSubscriber and the topic name as parameter.
29 \else
30 The built-in data-readers all belong to a built-in Subscriber. This subscriber can be
31 retrieved by using the method get_builtin_subscriber provided by the
32 DomainParticipant. The built-in DataReader objects can be retrieved by using the operation
33 lookup_datareader, with the Subscriber and the topic name as parameter.
34 \endif

84 Built-in entities have default listener settings as well. The built-in Subscriber and all
85 of its built-in Topics have nil listeners with all statuses appearing in their listener
86 masks. The built-in DataReaders have nil listeners with no statuses in their masks.
87 
88 The information that is accessible about the remote entities by means of the built-in
89 topics includes all the QoS policies that apply to the corresponding remote Entity.
90 The QoS policies appear as normal 'data' fields inside the data read by means of
91 the built-in Topic. Additional information is provided to identify the Entity and
92 facilitate the application logic.
93 
94 The tables below list the built-in topics, their names, and the additional information
95 (beyond the QoS policies that apply to the remote entity) that appears in the data
96 associated with the built-in topic.

98 ParticipantBuiltinTopicData {#DCPS_Builtin_Topics_ParticipantData}
99 ---------------------------
100 
101 The DCPSParticipant topic communicates the existence of DomainParticipants
102 by means of the ParticipantBuiltinTopicData datatype. Each
103 ParticipantBuiltinTopicData sample in a Domain represents a
104 DomainParticipant that participates in that Domain: a new
105 ParticipantBuiltinTopicData instance is created when a newly added
106 DomainParticipant is enabled, and it is disposed when that DomainParticipant is
107 deleted. An updated ParticipantBuiltinTopicData sample is written each
108 time the DomainParticipant modifies its UserDataQosPolicy.

123 TopicBuiltinTopicData {#DCPS_Builtin_Topics_TopicData}
124 ---------------------
125 
126 The DCPSTopic topic communicates the existence of topics by means of the
127 TopicBuiltinTopicData datatype. Each TopicBuiltinTopicData sample in
128 a Domain represents a Topic in that Domain: a new TopicBuiltinTopicData
129 instance is created when a newly added Topic is enabled. However, the instance is
130 not disposed when a Topic is deleted by its participant because a topic lifecycle is
131 tied to the lifecycle of a Domain, not to the lifecycle of an individual participant.
132 An updated TopicBuiltinTopicData sample is written each time a Topic modifies one or
133 more of its QosPolicy values.
134 
135 Information published in the DCPSTopicTopic is critical to the data distribution
136 service, therefore it cannot be disabled by means of the Domain/BuiltinTopics
137 element in the configuration file.

166 PublicationBuiltinTopicData {#DCPS_Builtin_Topics_PublicationData}
167 ---------------------------
168 
169 The DCPSPublication topic communicates the existence of datawriters by means
170 of the PublicationBuiltinTopicData datatype. Each
171 PublicationBuiltinTopicData sample in a Domain represents a datawriter in
172 that Domain: a new PublicationBuiltinTopicData instance is created when a
173 newly added DataWriter is enabled, and it is disposed when that DataWriter is
174 deleted. An updated PublicationBuiltinTopicData sample is written each
175 time the DataWriter (or the Publisher to which it belongs) modifies a QosPolicy that
176 applies to the entities connected to it. Also will it be updated when the writer looses
177 or regains its liveliness.
178 
179 \if isocpp2
180 The PublicationBuiltinTopicData Topic is also used to return data through the
181 \ref dds::sub::matched_publication_data(const dds::sub::DataReader<T>& dr, const ::dds::core::InstanceHandle& h)
182 "dds::sub::matched_publications_data" operation.
183 \else
184 The PublicationBuiltinTopicData Topic is also used to return data through
185 the get_matched_publication_data operation on the DataReader.
186 \endif

217 SubscriptionBuiltinTopicData {#DCPS_Builtin_Topics_SubscriptionData}
218 ----------------------------
219 
220 The DCPSSubscription topic communicates the existence of datareaders by
221 means of the SubscriptionBuiltinTopicData datatype. Each
222 SubscriptionBuiltinTopicData sample in a Domain represents a datareader
223 in that Domain: a new SubscriptionBuiltinTopicData instance is created
224 when a newly added DataReader is enabled, and it is disposed when that
225 DataReader is deleted. An updated SubscriptionBuiltinTopicData sample is
226 written each time the DataReader (or the Subscriber to which it belongs) modifies a
227 QosPolicy that applies to the entities connected to it.
228 
229 \if isocpp2
230 The SubscriptionBuiltinTopicData Topic is also used to return data through the
231 \ref dds::pub::matched_subscription_data(const dds::pub::DataWriter<T>& dw, const ::dds::core::InstanceHandle& h)
232 "dds::pub::matched_subscriptions_data" operation.
233 \else
234 The SubscriptionBuiltinTopicData Topic is also used to return data through
235 the get_matched_subscription_data operation on the DataWriter.
236 \endif

266 Other built-in topics {#DCPS_Builtin_Topics_Others}
267 ---------------------
268 
269 There are a number of other built-in topics that have not been mentioned. These
270 topics (e.g. DCPSDelivery, DCPSHeartbeat and potentially some others) are
271 proprietary and for internal use only. Users are discouraged from doing anything
272 with these topics, so as not to interfere with internal mechanisms that rely on them.
273 The structure of these topics may change without notification.
